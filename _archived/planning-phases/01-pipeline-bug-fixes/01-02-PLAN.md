<objective>
Fix PIPE-02: `public/live.json` must never display misleading stale counters between Playwright runs. Today, after a run ends the file retains `total: 2932, passed: 0` (or whatever the last run produced) until the next run's first `onTestEnd` mutates counters, creating a window where the dashboard shows a false "run just finished" state.

Purpose: The dashboard polls `live.json` every 3s. When users open the dashboard during an idle period or right before a new run begins, they currently see stale numbers that misrepresent the state. This blocks the downstream dashboard phases (unified view, freshness signals) from being trustworthy.

Output: (1) A modified `tools/live-reporter.js` where `onBegin()` replaces `this.state` entirely (instead of only setting `.total`), clearing leftover counters/recentTests from prior runs. (2) A modified `tools/run-live.sh` that writes a clean sentinel to `public/live.json` BEFORE invoking `npx playwright test`, so the ~1s window before the reporter's first save is also clean.
</objective>

In practice the reporter is re-instantiated per Playwright process (constructor re-runs), BUT the stale `public/live.json` FILE on disk is what the dashboard reads — and that file is only rewritten when `_save()` runs. If an observer polls the file AT the exact moment between script invocation and `onBegin`, the old file's stale values win.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Local test runner → filesystem | `live-reporter.js` writes `public/live.json` during runs |
| Local filesystem → GitHub Pages | `_pushToGitHub` PUTs the same content to GitHub via Contents API (requires `GITHUB_TOKEN` from env) |
| Dashboard browser → GitHub Pages | Read-only fetch of `public/live.json` every 3s |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05 | Info Disclosure | Sentinel JSON content in `run-live.sh` | accept | Sentinel contains only literal integers and literal "false" — no secrets. |
| T-01-06 | Tampering | Race between reporter `_save()` and manual sentinel write | accept | The pre-run sentinel write runs BEFORE `npx playwright test` boots the reporter process. No overlap. The EXIT trap runs AFTER reporter exits. |
| T-01-07 | DoS | GitHub API rate-limit from extra pushes | mitigate | The plan does NOT add any new `_pushToGitHub` calls. The reporter's existing throttle (`PUSH_INTERVAL_MS = 10_000`) continues to govern remote pushes. The pre-run sentinel is a LOCAL write only (no API call). |
| T-01-08 | Tampering | `GITHUB_TOKEN` exposure in new code | accept | No new code paths touch `GITHUB_TOKEN`. The pre-run sentinel is a local `cat >` heredoc with no network I/O. |
| T-01-09 | Info Disclosure | Previously-seen stale counters persisting in a visitor's browser after sentinel write | accept | Dashboard `pollLive()` fetches every 3s; stale state is gone within one poll cycle after reset. Acceptable UX degradation (max 3s window) — matches current design. |
