**Script structure:**

```
#!/usr/bin/env python3
"""

--dry-run: print table to stdout without writing or committing.

```

**Constants at top of file (D-01, D-02, D-03):**
```python
PLAYWRIGHT_MIN_PASS_PCT = 80
MAESTRO_MIN_HEALTH = 60
```

**Helper: load_manifest_data(project_root) -> tuple[dict, dict]**
- Opens `project_root / "public/manifest.json"`
- Iterates `manifest["reports"]` using `.get()` on all fields (defensive)
- Groups by `client_slug`, keeps entry with max `date` per slug
- Returns `(cowork_by_slug, maestro_by_slug)` where:
  - `cowork_by_slug`: only `platform == "b2b"` entries → `{slug: {verdict, score, date}}`
  - `maestro_by_slug`: only `platform == "app"` entries → `{slug: {score, date}}`
- Maestro N/A = slug absent from `maestro_by_slug` (D-08: determined dynamically, no hardcoded list)

**Helper: load_playwright_data(project_root) -> dict**
- `glob(str(project_root / "public/history/20*.json"))` sorted DESC
- Excludes files ending in "index.json"
- For each file: iterate `data.get("clients", {}).items()`
- Skip slug if already in `result` dict (we only want most recent)
- REAL RUN CHECK: `if passed + failed > 0` — add to result (with date = Path(f).stem)
- If `passed + failed == 0`: continue to next file (do NOT skip the slug — it may have real data in an older file)
- Returns `{slug: {tests, passed, failed, date}}` — only clients with real data

**Helper: classify_client(slug, pw_data, cowork_data, maestro_data) -> dict**
Evaluation order: BLOQUEADO first (per D-04). Use two flags: `blocked = False`, `all_green = True`.

Playwright block:
```python
pw = pw_data.get(slug)
if pw:
    pct = round(pw["passed"] / pw["tests"] * 100, 1) if pw["tests"] > 0 else 0
    if pct < PLAYWRIGHT_MIN_PASS_PCT:
        blocked = True
        reasons.append(f"Playwright {pct}% < {PLAYWRIGHT_MIN_PASS_PCT}%")
    # else: PW is green — all_green stays True
else:
    all_green = False
    reasons.append("Playwright: sin datos recientes")
```

Cowork block:
```python
cw = cowork_data.get(slug)
if cw:
    verdict = cw.get("verdict", "")
    if verdict == "BLOQUEADO":
        blocked = True
        reasons.append("Cowork: BLOQUEADO")
    elif verdict == "LISTO":
        pass  # green — all_green stays True
    else:
        # CON_CONDICIONES or anything else -> PENDIENTE (D-05)
        all_green = False
        reasons.append(f"Cowork: {verdict}" if verdict else "Cowork: sin veredicto")
else:
    all_green = False
    reasons.append("Cowork: sin reporte")
```

Maestro block (D-08: only if client has app entries):
```python
mt = maestro_data.get(slug)  # None = N/A client
if mt is not None:
    score = mt.get("score", 0)
    if score < MAESTRO_MIN_HEALTH:
        blocked = True
        reasons.append(f"Maestro score {score} < {MAESTRO_MIN_HEALTH}")
    # else: Maestro green — all_green stays True for this dimension
# If mt is None: N/A — does not affect blocked or all_green
```

Return:
```python
if blocked:
    return {"status": "BLOQUEADO", "reason": "; ".join(reasons)}
if all_green:
    return {"status": "LISTO"}  # No "reason" key for LISTO
return {"status": "PENDIENTE", "reason": reasons[0] if reasons else ""}
```

**Helper: build_all_slugs(pw_data, cowork_data, maestro_data) -> set**
Union of all slugs from the three sources. This ensures a client with Cowork data but no PW history is still evaluated.

   ```
   BLOQUEADO  prinorte       Maestro score 0 < 60
   PENDIENTE  bastien        Cowork: CON_CONDICIONES

   ────────────────────────────────────────────────────
   1 BLOQUEADO · 4 PENDIENTES · 1 LISTO
   → public/weekly-status.json updated
   ```

**Title:** `# QA LISTO — Criterios de Deploy-Readiness`

**Section: Qué es QA LISTO**
Brief explanation: the script `tools/evaluate-qa-listo.py` evaluates whether each client is ready to deploy based on three pipeline signals. It runs manually after fresh data is available and writes `public/weekly-status.json`.

**Section: Los tres estados**
| Estado | Significado | Color en dashboard |
|--------|-------------|-------------------|
| LISTO | El cliente puede deployarse con confianza | Verde |
| PENDIENTE | Falta información o hay condiciones — revisar antes de deployar | Ámbar |
| BLOQUEADO | Existe un bloqueo explícito — NO deployar sin resolver | Rojo |

**Section: Thresholds actuales**
Explicitly state (these are the values referenced by D-03):
- `PLAYWRIGHT_MIN_PASS_PCT = 80` — Un cliente necesita pasar al menos el **80%** de sus tests Playwright para no bloquear el deploy.
- `MAESTRO_MIN_HEALTH = 60` — El health score de Maestro debe ser al menos **60/100** para clientes con flujos APP.

**Section: Lógica de clasificación**
Describe evaluation in plain Spanish (no code):

1. **BLOQUEADO** (se evalúa primero — cualquiera de estas condiciones bloquea el deploy):
   - Playwright tiene datos y el % de tests pasados es **menor al 80%**
   - Cowork entregó veredicto `BLOQUEADO`
   - El cliente tiene flujos Maestro (APP) y el health score es **menor a 60**

2. **LISTO** (se asigna solo si TODAS las siguientes son verdad):
   - Playwright tiene datos y pasa **80% o más** de sus tests
   - Cowork entregó veredicto `LISTO`
   - Maestro health es **60 o más** (o el cliente no tiene flujos APP — N/A no penaliza)

3. **PENDIENTE** (caso por defecto — cuando no hay bloqueo explícito pero tampoco están todos los semáforos en verde):
   - Playwright aún sin datos recientes
   - Cowork entregó veredicto `CON_CONDICIONES` (puede deployarse con conciencia de las condiciones)
   - Cowork sin reporte aún
   - Mezcla de señales que no completa el criterio LISTO pero sin bloqueo

**Section: Maestro N/A**

N/A no afecta la clasificación — un cliente puede ser LISTO sin datos de Maestro. Esto se determina dinámicamente: cuando se agreguen flujos Maestro para un cliente, el script lo incluirá en la evaluación automáticamente.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Script → filesystem | Script reads public/manifest.json and public/history/*.json — local files written by our own tools |
| Script → git | Script shells out to git — runs as current user, pushes to main |
| weekly-status.json → dashboard | Dashboard reads the JSON and inserts reason into title attribute (XSS vector) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | public/weekly-status.json | accept | File is generated by script and committed via git — git history provides audit trail. Staging environment, not public-facing production. |
| T-05-02 | Information Disclosure | evaluate-qa-listo.py console output | accept | Prints client slugs and status — no credentials, no PII. Output is visible only to person running the script locally. |
| T-05-03 | Elevation of Privilege | subprocess git push | accept | Script runs as developer's local user with existing git credentials. No new privilege surface vs. existing publish-results.py pattern. |
| T-05-04 | Information Disclosure | reason field in weekly-status.json | mitigate | reason field is plain text (e.g. "Playwright 81% < 80%") — no secrets.
