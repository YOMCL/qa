## Summary

The single most important finding is that **parallelism is already configured** — `playwright.config.ts` has both `fullyParallel: true` and `workers: 4`. The question this phase needs to answer is: why is it still slow, and what prevents those 4 workers from being effective?

After reading all 6 config-validation spec files, the test isolation model is **fully safe for parallel execution**. Every single test creates its own `browser.newContext()` at the start and calls `context.close()` at the end. There are zero `beforeAll` hooks, zero `afterAll` hooks, zero `storageState` files, zero shared ports, and zero shared filesystem state between tests. Workers cannot step on each other.

The performance problem is not a parallelism-safety problem — it is a **test count × client count × slowness per test** problem. With 29 clients in `clients.ts` and ~65 conditional test definitions per spec file, Playwright generates hundreds of test instances. Many tests navigate to remote staging URLs (`{client}.solopide.me`), execute a full login flow, and wait on network timeouts. At `workers: 4` and 4-second average network round trips, the math still produces long runs. The correct intervention is: verify the config is correctly wired, measure real baseline, and validate that all 6 spec files are being picked up by the `b2b` project config.

**Primary recommendation:** Confirm `fullyParallel: true` is active and effective for the config-validation subdirectory, validate that workers are not being constrained by some other factor (such as the `b2b` project's `testDir` missing the subdirectory), and measure before/after. No structural code changes to test files are needed.

## Architecture Patterns

### How Playwright `fullyParallel` works

**What:** When `fullyParallel: true`, Playwright distributes individual tests (not files) across workers. Each worker gets its own browser instance. Tests within the same file that share a `test.describe` block can run on different workers.

**Current config:**
```typescript
// playwright.config.ts
fullyParallel: true,   // tests within a file run in parallel too (not just across files)
workers: 4,            // 4 browser processes run simultaneously
```

**What this means:** With `fullyParallel: true` and `workers: 4`, up to 4 tests run simultaneously at all times. This is already the correct configuration for config-validation.

### How the for-loop parametrization works with workers

```typescript
// cv-access.spec.ts — current pattern
for (const [key, client] of Object.entries(clients)) {
  test.describe(`Config validation — Access: ${client.name}`, () => {
    test(`${key}: anonymousAccess=${client.config.anonymousAccess}`, async ({ browser }) => {
      const context = await browser.newContext();
      // ...
      await context.close();
    });
  });
}
```

**What Playwright sees:** Each `test()` call inside the for-loop becomes an independent test entry. With 29 clients × ~3 conditional tests per spec = ~87 tests in cv-access alone. With `fullyParallel: true`, all 87 run across 4 workers.

**Source:** Playwright docs — `fullyParallel` applies to all tests in the suite, including those generated by loops. Verified against Playwright 1.x docs behavior (HIGH confidence).

### Test isolation model

Each test in every config-validation spec follows this pattern:
```typescript
async ({ browser }) => {
  const context = await browser.newContext();  // fresh context, no shared state
  const page = await context.newPage();
  // ... do test work ...
  await context.close();  // always closed
}
```

**Isolation guarantees:**
- No `storageState` is read or written
- No shared `page` fixture
- No `beforeAll` / `afterAll` hooks anywhere in config-validation
- No filesystem writes during tests
- No shared ports or sockets
- `context.close()` cleans up cookies, localStorage, session for that context

**Conclusion:** Tests are fully independent. Worker count can be increased safely.

### testDir coverage for config-validation

// playwright.config.ts

## Common Pitfalls

### Pitfall 1: Thinking `fullyParallel: false` (file-level) is the current state
**What goes wrong:** Team assumes tests within a file run sequentially because `fullyParallel` defaults to false. They add `test.describe.parallel()` to each spec.
**Why it happens:** Playwright's default is `fullyParallel: false`. But the config already explicitly sets `fullyParallel: true`.
**How to avoid:** Check `playwright.config.ts` first. Adding `test.describe.parallel()` on top of `fullyParallel: true` is redundant but harmless.
**Warning signs:** If tests within cv-cart.spec.ts are running sequentially, this flag is not being respected.

### Pitfall 2: Worker count too high causes staging rate limiting
**What goes wrong:** Increasing `workers` beyond 4-6 causes staging servers (`*.solopide.me`) to throttle or return 429s, producing false failures.
**Why it happens:** Each test hits a real staging B2B endpoint. With 10+ workers hammering the same Cloudflare-proxied endpoints simultaneously, rate limiting kicks in.
**How to avoid:** Keep `workers: 4` as default. Use `--workers=6` max for local runs. Never exceed 4 in CI unless staging rate limits are verified to be absent.
**Warning signs:** Tests that pass individually start failing with network timeouts or 429s when running in parallel.

### Pitfall 3: Assuming slow tests are a parallelism problem
**What goes wrong:** The suite takes 15+ minutes even with workers: 4. Team adds more workers, problem persists.
**Why it happens:** With 29 clients × ~4 avg tests per spec × 6 spec files = ~700 test instances. At 30s avg per test / 4 workers, that's still 87 minutes sequential equivalent = ~22 minutes at 4x. The ceiling at 4 workers with staging network calls is around 10-15 minutes.
**How to avoid:** Measure actual per-test duration. Long tests are a network/timing issue, not a parallelism config issue. The 50% reduction target is achievable if baseline was truly sequential.
**Warning signs:** `workers: 8` doesn't meaningfully reduce total time vs `workers: 4`.

### Pitfall 4: `addOneProductToCart` leaves cart state for the next test
**What goes wrong:** cv-cart.spec.ts tests that add products to the cart don't clean up. If the context is not closed, later tests in the same context see stale cart state.
**Why it happens:** `addOneProductToCart` mutates server-side cart state for the test user. The `context.close()` clears local browser state but NOT the server-side cart.
**How to avoid:** All cv-cart tests already call `context.close()` per-test. Each test re-creates the context and re-logs in, which re-initializes the cart view. Server-side cart state is per-user per-session, and since each test uses the same credentials, a cart added in test N may be visible in test N+1 IF they run on the same account concurrently.
**Warning signs:** Cart tests flake with unexpected item counts.
**Mitigation:** If cart state flakiness appears, add a `clearCartHelper` call at the start of cart tests (pattern already exists in `multi-client-auth.ts`).

### Pitfall 5: `workers: 4` in config is overridden by CLI flags
**What goes wrong:** Running `npx playwright test --workers=1` (e.g., in debug mode) and forgetting this overrides config. Success criteria check for `--workers=4` may pass locally but CI runs sequentially.
**How to avoid:** Confirm CI workflow uses `npx playwright test` without `--workers=1` override.

## Open Questions

1. **What is the actual baseline duration?**
   - What we know: 65 test definitions × 29 clients = many instances; `workers: 4` is set
   - What's unclear: No timing data was collected in Phase 2. We don't know if tests currently run at 4x or if some constraint is serializing them.
   - Recommendation: Wave 0 of the plan must measure baseline with `--workers=1` and current config before claiming improvement.

2. **Are any tests currently failing due to parallelism problems?**
   - What we know: No shared state in code, no beforeAll hooks
   - What's unclear: Server-side cart state conflicts under concurrent logins with same credentials across workers
   - Recommendation: Run cv-cart in isolation with `--workers=4` and check for cart-state flakiness.

3. **Does the CI workflow actually run Playwright?**
   - What we know: Only one GitHub Actions workflow exists (`deploy-qa-dashboard.yml`) and it only deploys the dashboard. No Playwright run workflow exists.
   - What's unclear: Whether tests are intended to run in CI or only locally.
   - Recommendation: The 50% improvement target should be measured locally. CI integration is out of scope for REQ-05 unless explicitly added.

4. **How many tests actually execute?**
   - What we know: 65 test() definitions across 6 files, but most are wrapped in `if (client.config.X !== undefined)` guards. With 29 clients having variable config flags, actual test count is less than 65 × 29 = 1885.
   - Recommendation: Run `npx playwright test b2b/config-validation/ --list` to get exact count before benchmarking.
